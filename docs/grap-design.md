# QEMU 虚拟显示设备与图形后端设计说明

## 1. QEMU 提供的虚拟显示设备

QEMU 可以模拟多类显示设备，它们面向的客户系统、驱动复杂度和功能能力并不相同。

### 1.1 `stdvga`

- `stdvga` 是最常见的传统 VGA 兼容设备
- 提供基本 VGA 能力，并在此基础上支持 VBE（VESA BIOS Extensions）风格的线性 framebuffer
- 兼容性较好，很多传统系统或早期引导阶段都容易接入
- 优点是资料多、模型成熟、调试门槛低
- 缺点是接口偏传统，功能现代化程度有限

### 1.2 `cirrus-vga`

- `cirrus-vga` 模拟较早期的 Cirrus Logic 显卡
- 历史上常用于兼容老系统
- 对一些旧版操作系统兼容较好
- 但它已经不是现代系统首选，功能和生态都不如更新方案

### 1.3 `vmware-svga`

- `vmware-svga` 提供 VMware 风格的图形设备接口
- 比传统 VGA 更现代，支持更灵活的显示模式
- 常见于支持 SVGA 驱动的客户系统
- 对教学和现代化图形实验来说，比纯 VGA 更接近“专用图形设备”模型

### 1.4 `bochs-display` / `ramfb`

- `bochs-display` 常被视为一种简单、清晰、适合虚拟机场景的线性 framebuffer 显示设备
- `ramfb` 更进一步，核心思想是“给客户系统一块 framebuffer 内存，写进去就能显示”
- 这类方案非常适合内核早期 bring-up、实验系统、教学用途
- 优点是驱动模型简单，容易完成“初始化 + 写像素 + 显示输出”
- 缺点是功能较弱，不适合复杂桌面图形加速

### 1.5 `virtio-vga`

- `virtio-vga` 是把 VGA 兼容入口和 virtio 现代虚拟设备能力结合起来的方案
- 既保留一定兼容性，又支持 virtio 的现代化抽象
- 对需要兼顾启动兼容性和后续扩展能力的系统比较合适

### 1.6 `virtio-gpu`

- `virtio-gpu` 是当前虚拟化环境中非常重要的现代显示设备
- 它不再局限于“传统 VGA 寄存器 + framebuffer”模式，而是通过 virtio 队列与宿主交互
- 支持资源创建、2D 绘图传输、scanout 绑定等机制
- 更适合现代内核和桌面系统，也更利于后续扩展 Wayland、DRM/KMS 一类图形栈
- 缺点是驱动实现复杂度明显高于简单 framebuffer

### 1.7 `qxl`

- `qxl` 常与 SPICE 远程显示方案配合使用
- 它强调虚拟桌面和远程图形体验
- 在需要远程桌面协议支持时比较有价值
- 但对轻量教学内核而言，驱动复杂度通常偏高

## 2. QEMU 常见图形后端

图形后端决定“QEMU 最终如何把客户系统生成的图像展示出来”。显示设备负责向 QEMU 提供图像内容，图形后端负责把这些内容显示到宿主机窗口或远程客户端。

### 2.1 `gtk`

- `gtk` 是 QEMU 最常见的本地图形窗口后端之一
- 会在宿主机上打开一个 GTK 窗口显示客户系统画面
- 使用方便，适合日常开发和调试
- 常见启动方式：

```bash
qemu-system-riscv64 -display gtk
```

### 2.2 `sdl`

- `sdl` 也是常见的本地图形后端
- 适合跨平台窗口显示
- 与 `gtk` 类似，主要用于本机直接观察图形输出

### 2.3 `cocoa`

- `cocoa` 是 macOS 上的本地图形后端
- 主要用于 Apple 平台宿主机
- 在 Linux 实验环境中通常不作为首选

### 2.4 `vnc`

- `vnc` 是远程图形后端
- QEMU 不一定在本地弹窗，而是把显示内容通过 VNC 服务导出
- 用户可以用 VNC 客户端远程连接观察系统图形输出
- 适合服务器、远程开发和 CI 环境

### 2.5 `spice`

- `spice` 是更完整的远程图形协议方案
- 通常与 `qxl` 或其他适配设备联合使用
- 除了图形显示，还能支持更丰富的远程交互能力
- 适合虚拟桌面类场景

### 2.6 `egl-headless`

- `egl-headless` 适合无窗口、无本地桌面的宿主环境
- 常用于图形渲染测试、服务器环境或与其他远程展示链路配合
- 重点是“渲染存在，但不一定直接弹出本地窗口”

### 2.7 `none`

- `none` 表示不启用常规图形窗口输出
- 常配合串口日志或专门调试路径使用
- 如果没有额外远程显示配置，用户通常看不到图形画面

## 3. 客户系统如何操作这些虚拟显示设备，并在图形后端上显示图形

整个链路可以概括为：

`客户系统驱动 -> 虚拟显示设备接口 -> QEMU 设备模型 -> QEMU 图形后端 -> 宿主机窗口或远程客户端`

### 3.1 客户系统侧的基本操作流程

无论是哪种虚拟显示设备，客户系统通常都要完成以下工作：

1. 识别设备
   - 通过 PCI、MMIO 或平台总线发现显示设备
   - 判断设备类型，例如传统 VGA、Bochs framebuffer、virtio-gpu
2. 初始化设备
   - 配置分辨率、像素格式、stride
   - 获取 framebuffer 地址，或建立 virtqueue / command queue
3. 建立可写显示缓冲区
   - 对简单 framebuffer 设备，通常直接拿到一块线性显存
   - 对 `virtio-gpu`，通常需要先创建 resource，再把内存传给设备
4. 输出图形
   - 最简单方式是直接写 framebuffer 内存
   - 更复杂的设备会通过命令队列提交更新、flush 或 scanout 操作
5. 通知刷新
   - 有些设备只要写显存就会被 QEMU 检测到脏区
   - 有些设备需要显式 flush，告诉设备“这一块图像已经更新”

### 3.2 对简单 framebuffer 设备的操作方式

以 `stdvga`、`bochs-display`、`ramfb` 这类设备为例：

- 驱动先读取寄存器或设备配置，得到宽度、高度、每像素字节数和 framebuffer 地址
- 然后把 framebuffer 映射到内核可访问地址空间
- 像素坐标 `(x, y)` 会被转换成线性偏移

```text
offset = y * stride + x * bytes_per_pixel
pixel_addr = framebuffer_base + offset
```

- 驱动把颜色值写到对应位置后，QEMU 的设备模型就能看到 framebuffer 内容变化
- QEMU 再把变化后的像素交给 `gtk`、`sdl`、`vnc`、`spice` 等图形后端显示

这类模型最适合本项目这种“内核直接写像素”的实验需求。

### 3.3 对 `virtio-gpu` 的操作方式

`virtio-gpu` 的流程比简单 framebuffer 更现代，也更复杂：

- 客户系统先初始化 virtio 设备
- 建立控制队列、可能还包括光标队列
- 通过命令创建 2D resource
- 把一块客户内存与 resource 关联
- 设置 scanout，把该 resource 绑定到显示输出
- 每次绘制后执行 transfer / flush，让 QEMU 更新显示内容

这意味着客户系统不是“只写显存就结束”，而是“写显存 + 通过 virtio 协议告诉设备刷新哪部分图像”。

### 3.4 图形后端如何参与显示

图形后端本身并不直接决定客户系统怎么编程设备，它负责最后一步展示：

- 如果后端是 `gtk` 或 `sdl`，QEMU 会在宿主机本地创建窗口
- 如果后端是 `vnc`，QEMU 会把画面编码后通过 VNC 导出
- 如果后端是 `spice`，QEMU 会通过 SPICE 协议提供远程桌面能力

因此，客户系统真正面对的是“虚拟显示设备”，而不是直接面对 `gtk` 或 `vnc`。图形后端更像是显示结果的出口。

## 4. 3 条建议选择的虚拟显示设备和图形后端组合

下面给出三组更有针对性的推荐。

### 方案一：`bochs-display` 或 `ramfb` + `gtk`

适合场景：

- 教学实验
- 自研内核早期图形 bring-up
- 需要尽快看到“写像素就出图”的效果

推荐理由：

- 设备模型简单
- 驱动实现成本低
- `gtk` 本地窗口调试直观
- 最适合 framebuffer 基础实验

如果目标是像本项目这样在 QEMU 中实现基础图形输出，这是最推荐的入门组合。

### 方案二：`stdvga` + `sdl`

适合场景：

- 需要较强兼容性
- 希望沿用传统 VGA/VBE 方式
- 需要和较多历史资料、已有实验代码保持一致

推荐理由：

- 传统 VGA 生态成熟
- `sdl` 本地显示简单稳定
- 适合“兼容优先”的课程实验和原型开发

这组方案的优势是资料多、理解路径清晰，但现代扩展性不如 virtio 系列。

### 方案三：`virtio-gpu` + `vnc` 或 `spice`

适合场景：

- 现代虚拟化图形栈实验
- 远程访问需求明显
- 后续想向 DRM/KMS、桌面系统或更复杂图形协议扩展

推荐理由：

- `virtio-gpu` 是现代虚拟 GPU 的主流方向
- `vnc` 和 `spice` 适合远程开发、远程调试和服务器环境
- 更利于扩展复杂图形系统

这组方案能力最强，但驱动实现和调试复杂度也最高。

## 5. 结合本项目的建议

考虑到本项目是 `lbl-tg-rcore-tutorial-vga`，重点是 QEMU 中的基础图形输出与 framebuffer 访问，因此建议优先级如下：

1. 首选：`bochs-display` 或 `ramfb` + `gtk`
2. 次选：`stdvga` + `gtk` 或 `sdl`
3. 进阶：`virtio-gpu` + `vnc`

原因是：

- 本项目更强调“驱动简单、容易验证、便于教学”
- 简单 framebuffer 设备最适合直接映射显存并写像素
- 本地窗口后端最利于观察调试结果
- `virtio-gpu` 更适合作为后续增强方向，而不是最初版本
